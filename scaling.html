縮放 (Scaling)

對基因進行 Z-score 標準化，讓每個基因在 PCA 中擁有平等的「發言權」。

Z-score normalization gives every gene equal "voice" in PCA analysis.

概覽

一、為什麼要 Scaling？

經過 Normalization 之後，不同基因的表達範圍仍差異極大——某些結構/核醣體基因可能均值仍在 5–8 區間，而轉錄因子或 marker 可能均值僅 0.1–0.5。如果直接送 PCA，前幾個主成分將被「高表達 + 高方差」的少數基因主導，真正帶有細胞型別訊號的 marker 基因被淹沒。

Scaling 在做什麼？對每一個基因做 Z-score 標準化（減均值、除以標準差），讓所有基因均值 0、變異數 1 的同一尺度上競爭。這樣 PCA 才是真正計算「協方差結構」，而不是「誰最亮」。

類比：想像你在比較 100 位歌手的「音域變化」。如果有人用日語唱、有人用英語唱、聲量還相差 10 倍，光看絕對音量比較不公平。先把每個人的音量都標準化（均值 0、變異 1）後再比「相對音域形狀」，才能找出歌唱風格的真實差異。Scaling 就是 PCA 的「音量標準化」。

After Normalization, different genes still have vastly different expression ranges — some structural/ribosomal genes have means of 5–8, while transcription factors or markers may average 0.1–0.5. Feeding this into PCA directly lets the first few PCs be dominated by a few "bright + variable" genes, while real cell-type markers are buried.

What does Scaling do? Z-score each gene (subtract mean, divide by SD), placing all genes on the same scale: mean 0, variance 1. Now PCA truly captures covariance structure, not "which gene is brightest".

Analogy: Imagine comparing 100 singers' "range variation". If some sing in Japanese, some in English, with volumes differing 10×, absolute-volume comparison is unfair. First normalize everyone's volume (mean 0, var 1), then compare "relative pitch shape" — that's how you find real singing-style differences. Scaling is PCA's volume normalization.

📐 數學公式

zij = (xij − μi) / σi

xij = 第 i 個基因在第 j 個細胞的 log-normalized 表達
μi, σi = 該基因跨所有細胞的均值與標準差
注意：是「對基因」做 Z-score，不是對細胞。

xij = log-normalized expression of gene i in cell j
μi, σi = mean and SD across all cells for that gene
Note: Z-score is per-gene, not per-cell.

🆚 與 Normalization 差異

Normalization：「跨基因」操作——校正每個細胞的深度。
Scaling：「跨細胞」操作——讓每個基因同尺度。
兩者方向正交、互補不可替代。

Normalization: "across genes" — corrects each cell's depth.
Scaling: "across cells" — equalizes each gene's scale.
They are orthogonal and complementary.

數學細節

二、為什麼 PCA 之前一定要中心化？

PCA 找的是變異最大的方向，數學上是協方差矩陣的特徵向量。協方差定義是 cov(X, Y) = E[(X − μX)(Y − μY)]——必須先減均值，否則計算出來的不是「變異」而是「原點偏離量」，PC1 會直接指向資料雲的「重心」方向，不帶任何結構訊息。

除均值還要除 SD 的原因：若不除，variance 大的基因會自動贏——而 variance 又跟 mean 高度相關（單細胞數據）。Scaling 把兩者都歸一化，留下的才是「不同細胞之間誰一起高、誰一起低」的結構訊號。

PCA before scaling: 第一個 PC ≈ 高均值高方差基因的方向（無生物意義）
PCA after scaling: 第一個 PC ≈ 細胞型別 / 狀態間的最大協變方向 ✓

Seurat 的選擇：預設只 scale HVG（節省時間 + 記憶體），存於 scale.data。視覺化 heatmap 若想看 non-HVG，需顯式指定 features=... 重新 scale 才有完整 scale.data。

Scanpy：sc.pp.scale() 對 .X 全部基因做 in-place 標準化，並支援 max_value=10 把極端值 clip 到 ±10（避免 outlier 主導 PCA）。

PCA finds directions of maximum variance, mathematically the eigenvectors of the covariance matrix. Covariance is cov(X, Y) = E[(X − μX)(Y − μY)] — mean must be subtracted; otherwise we compute "deviation from origin" rather than variance, and PC1 just points at the data cloud's centroid with no structural information.

Why also divide by SD: without it, genes with larger variance automatically win — and variance correlates strongly with mean in single-cell data. Scaling normalizes both, leaving only the "who-co-varies-with-whom" structural signal.

PCA before scaling: PC1 ≈ direction of high-mean high-variance genes (no biology)
PCA after scaling: PC1 ≈ largest covariation between cell types / states ✓

Seurat default: only scales HVG (saves time + memory), stored in scale.data. To view non-HVG in heatmaps, explicitly pass features=... to re-scale and produce a complete scale.data.

Scanpy: sc.pp.scale() standardizes all genes in .X in place; supports max_value=10 to clip extreme values to ±10 (prevents outliers from dominating PCA).

💡
記憶口訣：Normalization 校正「行（細胞）」、Scaling 校正「列（基因）」、Regression 校正「特定變數」。三者方向正交，可同時使用但不要弄混順序：normalize → HVG → scale (+ regress) → PCA。 Mnemonic: Normalization corrects "rows (cells)", Scaling corrects "columns (genes)", Regression corrects "specific variables". Orthogonal — can be combined but never mix the order: normalize → HVG → scale (+ regress) → PCA.

程式碼

五、實作範例

# ===== Seurat: 三種常見寫法 =====

# A. 最基本：scale 預設 HVG
pbmc <- ScaleData(pbmc)
# 結果存於 pbmc[["RNA"]]@scale.data

# B. 想 heatmap 看全基因：scale 全部
all.genes <- rownames(pbmc)
pbmc <- ScaleData(pbmc, features = all.genes)

# C. 進階：回歸技術變數
pbmc <- CellCycleScoring(pbmc, s.features = s.genes, g2m.features = g2m.genes)
pbmc <- ScaleData(pbmc,
  vars.to.regress = c("percent.mt", "S.Score", "G2M.Score"),
  features = VariableFeatures(pbmc))

# 確認 scale.data 的尺度
apply(GetAssayData(pbmc, "scale.data")[1:5,], 1, mean)  # 應接近 0
apply(GetAssayData(pbmc, "scale.data")[1:5,], 1, sd)    # 應接近 1

# ⚠ 使用 SCTransform 者不需再呼叫 ScaleData——SCT@scale.data 已存在
import scanpy as sc

# ===== Scanpy: 三種常見寫法 =====

# A. 最基本：clip 至 ±10 防 outlier
sc.pp.scale(adata, max_value=10)

# B. 進階：先 regress_out，再 scale
sc.pp.regress_out(adata, ["pct_counts_mt", "total_counts"])
sc.pp.scale(adata, max_value=10)

# C. 細胞週期回歸（可選）
sc.tl.score_genes_cell_cycle(adata,
    s_genes=s_genes, g2m_genes=g2m_genes)
sc.pp.regress_out(adata, ["S_score", "G2M_score"])
sc.pp.scale(adata, max_value=10)

# 驗證
import numpy as np
np.mean(adata.X, axis=0)[:5]   # ≈ 0
np.std(adata.X, axis=0)[:5]    # ≈ 1

# ⚠ scale 之後一定要先 PCA，不要再做 normalize/log
sc.tl.pca(adata, n_comps=50)
最佳實務

六、最佳實務

🎚️ max_value=10

把 Z-score 超過 ±10 的值 clip 掉。避免極少數細胞 outlier（> 50 SD）綁架 PCA。Seurat scale.max=10、Scanpy max_value=10 皆為公認標準。

Clip Z-scores beyond ±10. Prevents a tiny outlier (> 50 SD) from hijacking PCA. Both Seurat scale.max=10 and Scanpy max_value=10 are standard.

🛑 整合前不要 scale

Harmony / RPCA / scVI 流程中，scaling 通常在整合之後進行，或由整合演算法內部完成（scVI 在隱空間直接消除批次）。預先 scale 會固化批次差異。

For Harmony / RPCA / scVI workflows, scaling typically happens after integration, or is handled internally (scVI removes batch in latent space). Pre-scaling locks in batch differences.

🧬 細胞週期看研究目的

研究發育、分化、增殖細胞群：不要回歸週期分數；它們本身就是生物學主軸。研究組織異質性、罕見細胞型別：可回歸以避免 cluster 按週期切分。

Studying development, differentiation, proliferation: don't regress cycle scores — they are biological axes. Studying tissue heterogeneity, rare types: regress to avoid clusters split by cycle.

💾 記憶體不足對策

scale.data 是 dense matrix，10 萬細胞 × 2 萬基因約 15 GB。對策：(1) 預設 HVG only；(2) Seurat v5 用 BPCells on-disk；(3) 改用 Scanpy + dask。

scale.data is dense; 100k cells × 20k genes ≈ 15 GB. Mitigation: (1) HVG only by default; (2) Seurat v5 BPCells on-disk; (3) Scanpy + dask.

📈 驗證 scale

挑 5 個基因檢查 row mean ≈ 0、row sd ≈ 1。若不符通常代表 scale 沒跑或被覆蓋。Seurat 也提供 rowMeans 視覺化。

Check row mean ≈ 0 and row SD ≈ 1 on 5 genes. Mismatch usually means scaling wasn't run or was overwritten. Seurat offers rowMeans for visualization.

🚦 SCTransform 流程

SCTransform 已內建 scaling（產出 Pearson residuals），不要再呼叫 ScaleData。直接 RunPCA。若 RNA assay 也想用 DE，請另外切回去做 NormalizeData，不需 scale。

SCTransform builds in scaling (Pearson residuals). Do not call ScaleData; go directly to RunPCA. For RNA-assay DE, switch back and run NormalizeData; no scaling needed.

陷阱

七、常見陷阱

⚠️
Scaling 階段最常見的 5 個錯誤：
① 忘了 scale 就跑 PCA——Seurat RunPCA 會警告並只用未 scaled 數據，Scanpy sc.tl.pca 卻不會警告，直接得到「PC1 = library size」的災難結果。
② vars.to.regress 把生物訊號回歸掉——研究 T cell exhaustion 卻回歸了 G2M.Score。
③ SCTransform 後又呼叫 ScaleData——雙重 scale 把 Pearson residual 進一步壓縮，分群崩潰。
④ 整合前 scale——固化批次差異，Harmony 拉不開。
⑤ scale.data 找不到（在重新讀 RDS 後）——scale.data 不被 saveRDS 完整保留，需重跑 ScaleData。
Top 5 Scaling-stage mistakes:
① Running PCA without scaling — Seurat RunPCA warns and falls back; Scanpy sc.tl.pca doesn't warn and silently gives the disaster "PC1 = library size".
② vars.to.regress wipes biological signal — studying T-cell exhaustion but regressing G2M.Score.
③ ScaleData after SCTransform — double scaling compresses Pearson residuals further; clustering breaks.
④ Scaling before integration — bakes in batch effects; Harmony can't separate.
⑤ scale.data missing after re-loading RDS — scale.data isn't fully preserved by saveRDS; re-run ScaleData.

🚨 Scanpy PC1=深度

原因：未跑 sc.pp.scale() 或順序錯誤。
修正：確保流程是 normalize → log1p → HVG → scale → PCA。檢查 adata.X 的 row means 是否約等 0。

Cause: sc.pp.scale() not called or wrong order.
Fix: Ensure order: normalize → log1p → HVG → scale → PCA. Check adata.X row means ≈ 0.

🚨 回歸後 cluster 變少

原因：vars.to.regress 把生物變異一起移除（最常見是錯回歸 cell cycle）。
修正：移除該變數重試；或先 PCA 不回歸看清楚分群，再決定要不要回歸。

Cause: vars.to.regress removed biological variance (often cell cycle).
Fix: Remove that variable; or PCA without regression first, then decide if regression is needed.

🚨 scale.data 缺基因

原因：預設只 scale HVG，DoHeatmap 想看非 HVG marker 會報空白。
修正：ScaleData(obj, features=rownames(obj)) 跑一次（記憶體允許下）。

Cause: Default scales only HVG; DoHeatmap on non-HVG markers gives blanks.
Fix: Run ScaleData(obj, features=rownames(obj)) once (memory permitting).

🚨 SCT scale.data 不見

原因：SCTransform 預設只在 var.features 上產 scale.data。
修正：顯式 SCTransform(..., return.only.var.genes=FALSE)，但會大幅增加記憶體。

Cause: SCTransform produces scale.data only for var.features by default.
Fix: Pass return.only.var.genes=FALSE; memory usage will spike.

🚨 ScaleData 很慢

原因：vars.to.regress 啟用後使用 linear model，大樣本極耗時。
修正：(1) Seurat 設 model.use='linear' 已最快；(2) 不要對全基因加回歸；(3) 改用 SCTransform 內建回歸。

Cause: vars.to.regress runs linear models — slow on large data.
Fix: (1) model.use='linear' is already fastest; (2) don't regress on all genes; (3) use SCTransform's built-in regression instead.

🚨 重讀檔 scale.data 空

原因：有些 saveRDS 工具或 conversion 流程不保留 dense scale.data。
修正：讀檔後重跑 ScaleData()，或改存 .qs 格式（qs::qsave 較完整）。

Cause: Some saveRDS or conversion flows drop dense scale.data.
Fix: Re-run ScaleData() after reload, or use qs::qsave for fuller preservation.
